Debugging a bad compile

Every time you save, pause, or enable a piston, PistonCore compiles it and pushes the result into Home Assistant. When that goes wrong, the problem is announced in exactly two places: the status pill on the piston's row on the landing page, and a banner on the piston's own status screen. Everything below is about what to do after you've seen one of those.

What each status means

StatusWhat happenedYour first move
deployed Compiled, written into HA, HA's config check passed, and the automation was verified present and enabled after the reload. This is the only green. Nothing — but see "when it compiles fine and still misbehaves" below.
compile error The piston uses something the compiler can't build yet, or a device that no longer resolves. The message names the exact statement and construct. Nothing in HA was touched — the previous working version keeps running. Read the message. "Not compiled yet" means scheduled work, not a broken piston — the piston itself is fine and saved. A device error means re-pick that device in the editor and save.
HA rejected The compiler produced YAML that Home Assistant refused — this is a compiler or template bug, not a mistake in your piston. HA's own error text is quoted in the message. The previous good version was automatically restored, so nothing broke. This is exactly the case the evidence folder below exists for. Collect the evidence package and get it fixed (bottom of this page).
needs PyScript The piston is valid but uses features beyond plain HA automations. It's routed to the PyScript compiler, which isn't built yet. Wait for the PyScript band, or simplify the piston if you want it live now. The reasons listed name the specific features responsible.
paused You paused it — its automation is disabled inside HA (or removed, if HA couldn't be reached to disable it; the message says which). Nothing. Enable recompiles fresh and turns it back on.

The evidence folder

Every compile keeps a timestamped copy of exactly what it produced, in the PistonCore data folder (on the machine running PistonCore — the folder you mapped to /data):

PathWhat's in it
compile_debug/<piston-id>/ One .yaml per compile, newest 25 kept. If a compile crashed partway, a .partial.yaml holds whatever it built before dying — half-finished output is a clue, not garbage. Nothing here is ever deleted by pausing or recompiling.
compile_status.json The latest full status record for every piston — the same message the banner shows, plus the file it deployed, the automation ids, and how far the deploy got (config check, reload, verification).
pistoncore/automations/ (inside HA's config folder) The live compiled files HA actually runs. Overwritten on every recompile — history lives in compile_debug/, not here.

The folder name is the piston's internal id; the deployed filename in the banner ends with the first 8 characters of it, which is how you match them up.

When it compiles fine and still misbehaves

A green pill means HA accepted the automation — not that the logic is right. Two kinds of problems slip through every check: a template that only fails when it actually runs, and logic that's valid but wrong. For those, use Home Assistant's own trace view: Settings → Automations, find PistonCore: <piston name>, open it, and click Traces after it fires (or fails to). The trace shows the exact path taken — which trigger fired, what every condition evaluated to, and where it stopped. It's the closest thing HA has to webCoRE's own trace screen.

Then compare what the trace did against the newest file in that piston's compile_debug/ folder. If an older compile behaved correctly, comparing the two files usually points straight at what changed.

Getting it fixed: the evidence package

Three things make any bad compile diagnosable — by an AI chat, a bug report, or future-you. Collect all three:

  1. The piston JSON — Import/Export page → Export → copy. This is the compiler's exact input.
  2. The newest artifact from compile_debug/<piston-id>/ (the .yaml or .partial.yaml). This is the compiler's exact output.
  3. The banner message, word for word — when HA rejected the file, it contains HA's own objection, down to the line number.

Paste all three into a chat and say what the piston was supposed to do. Input, output, and the verdict — that's everything a compiler bug needs to be found.

Safety facts worth knowing

  • A failed compile never loses your piston and never touches the working version already running in HA.
  • If a new compile fails HA's config check, the previous good file is restored automatically before HA reloads anything.
  • Pause always works: if HA can't be reached to disable the automation, the file is removed instead. The paused message tells you which happened.
  • The compiler never writes into your piston — the JSON you author is read-only to it, always.
