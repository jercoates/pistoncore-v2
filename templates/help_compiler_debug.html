Debugging a bad compile

Every time you save, pause, or enable a piston, PistonCore compiles it and pushes the result into Home Assistant. When that goes wrong, the problem is announced in exactly two places: the status pill on the piston's row on the landing page, and a banner on the piston's own status screen. Everything below is about what to do after you've seen one of those.

What each status means

StatusWhat happenedYour first move
deployed Compiled, written into HA, HA's config check passed, and the automation was verified present and enabled after the reload. This is the only green. Nothing — but see "when it compiles fine and still misbehaves" below.
compile error The piston uses something the compiler can't build yet, or a device that no longer resolves. The message names the exact statement and construct. Nothing in HA was touched — the previous working version keeps running. Read the message. "Not compiled yet" means scheduled work, not a broken piston — the piston itself is fine and saved. A device error means re-pick that device in the editor and save.
HA rejected The compiler produced YAML that Home Assistant refused — this is a compiler or template bug, not a mistake in your piston. HA's own error text is quoted in the message. The previous good version was automatically restored, so nothing broke. Open Diagnostics, click Copy debug info on that piston, and report it — nothing else to collect.
needs PyScript The piston is valid but uses features beyond plain HA automations, so it compiles to a PyScript module instead — which needs the PyScript integration installed in HA. Install PyScript from HACS, then re-save the piston. Diagnostics tells you whether it's installed.
paused You paused it — its automation is disabled inside HA (or removed, if HA couldn't be reached to disable it; the message says which). Nothing. Enable recompiles fresh and turns it back on.

Start here: the Diagnostics page

Diagnostics (also linked from Settings) does the digging for you. It checks this instance end to end — Home Assistant connection, write access, PyScript, the configuration.yaml lines — and tells you the fix for anything failing. Below that, every piston is listed with its real compile result; failing ones are already expanded.

For any piston you can, without leaving the browser:

  • Read the exact error — the same message the banner shows, with the statement number it came from.
  • View the generated code — the actual YAML or Python PistonCore produced, plus the last several versions it kept, so you can compare a working compile against a broken one.
  • Copy debug info — one click puts the complete evidence package on your clipboard: the compile status, the generated code, and the piston's own JSON. Paste it into an AI chat or a bug report and nothing else is needed.

Fixing the common errors

What the message saysWhat to do
“…not compiled yet” / “requires PyScript” The construct is real but this band can't build it. If PyScript isn't installed, install it (HACS → “pyscript”) and re-save — most of these disappear. If it is installed, the message names a construct that genuinely isn't built yet; simplify that statement or wait for it.
“global device variable '@X' not found” or “has no devices assigned” Open the piston, and in the editor's Global variables panel create/assign that variable's devices. Names are case-sensitive — @Speakers_All and @speakers_all are different variables. The message suggests a near-match when it finds one.
“device … has no binding for attribute 'X'” The device you picked doesn't expose that attribute in Home Assistant. Open the piston and re-pick the device (or the attribute) from the editor's list — the picker only offers what really exists.
“HA rejected the compiled YAML” / “PyScript did not load the module” A compiler bug, not your piston — and the previous working version was restored automatically. Home Assistant's own complaint is quoted in the message. Use Copy debug info on the Diagnostics page and report it; that bundle contains everything needed to fix it.
“no HA service mapping for command 'X' on domain 'Y'” That webCoRE command has no Home Assistant equivalent wired yet for that kind of device. The mappings live in editable data files, so this is usually a small addition rather than a code change — worth reporting with the bundle.
“system variable '$X' has no HA equivalent (external data feed)” The piston reads a SmartThings-era weather/HTTP feed that doesn't exist in HA. Replace it with a Home Assistant sensor (weather integration, REST sensor) and reference that device instead.

When it compiles fine and still misbehaves

A green pill means HA accepted the automation — not that the logic is right. Two kinds of problems slip through every check: a template that only fails when it actually runs, and logic that's valid but wrong. For those, use Home Assistant's own trace view: Settings → Automations, find PistonCore: <piston name>, open it, and click Traces after it fires (or fails to). The trace shows the exact path taken — which trigger fired, what every condition evaluated to, and where it stopped. It's the closest thing HA has to webCoRE's own trace screen.

Then compare what the trace did against the generated code — Diagnostics → that piston → View generated code. Older versions are kept in the same dropdown, so if it worked last week, comparing the two usually points straight at what changed.

Getting help on something specific

Use Copy debug info on the Diagnostics page, paste the result into a chat or issue, and say what the piston was supposed to do. That bundle already contains the compiler's input (your piston JSON), its output (the generated code), and the verdict (the status record with any Home Assistant complaint) — input, output and error together, which is everything a diagnosis needs.

Safety facts worth knowing

  • A failed compile never loses your piston and never touches the working version already running in HA.
  • If a new compile fails HA's config check, the previous good file is restored automatically before HA reloads anything.
  • Pause always works: if HA can't be reached to disable the automation, the file is removed instead. The paused message tells you which happened.
  • The compiler never writes into your piston — the JSON you author is read-only to it, always.
