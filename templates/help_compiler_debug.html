When pistons don't compile

Start at Diagnostics — it runs the checks below for you and shows every piston's real result. This page explains what those results mean and, importantly, how to extend the compiler yourself when it hits something it doesn't know yet.

Situation 1 — nothing works yet (first run)

If every piston is red or stuck on “not compiled”, the problem is almost never the pistons. It's plumbing, and it fails in a fixed order — each step depends on the one above it:

  1. Home Assistant connection. Without it nothing else can even be checked. Settings → URL + long-lived token.
  2. Write access. PistonCore has to put files inside HA's config folder — a mounted path or HA's Samba share. Settings → write target → Test write target until it passes. Until it does, every compile fails at the last step.
  3. configuration.yaml include lines. HA only loads PistonCore's folders if its config points at them. Settings → Check configuration.yaml (shows the exact lines, backs the file up first), then reload or restart HA.
  4. PyScript. Simple pistons don't need it. Anything with formulas, loops, switches, variables, event blocks or computed text does. Install from HACS (search “pyscript”), then re-save.
  5. Globals and devices. Imported pistons reference global variables by name — the devices live in your instance, not in the piston file. Create them in the editor's Global variables panel using the same names (case-sensitive) and assign devices.

Diagnostics checks 1–4 directly and names the one that's broken, so you don't have to guess which layer failed.

Situation 2 — it worked, and now it doesn't

Something changed. Work from what changed:

  • Compare the generated code. Diagnostics → the piston → View generated code. The last several compiles are kept in the dropdown; diffing the last good one against the current one usually shows exactly what the compiler started doing differently.
  • “HA rejected the compiled YAML” / “PyScript did not load” means the compiler produced something Home Assistant refused — a compiler bug, not your piston. HA's own complaint is quoted in the message, and the previously working version was restored automatically, so nothing is down while you sort it out.
  • An entity was renamed or removed in HA. The piston still points at the old one — re-pick that device in the editor.
  • It compiles but behaves wrong — see Situation 4.

Situation 3 — the compiler doesn't know something yet

This is the most common real error and the most fixable. Messages like “no HA service mapping for command 'setColorTemperature' on domain 'light'” or “expression function 'formatduration()' not compiled yet” mean there's simply no entry for that thing yet.

Most of that knowledge lives in plain editable files, not in code — deliberately, so it can grow without a development session:

FileHolds
templates/compiler/yaml/classic/command_maps.json webCoRE commands → HA services (turn on, set level, set colour, play track, lock…) including the data each call sends.
templates/compiler/yaml/classic/value_maps.json Attribute values (motion active → on), alarm-status commands, colour names, system-variable values.
templates/compiler/pyscript/2.x/expr_runtime.py.j2 Every webCoRE expression function (format, indexOf, date maths…) as ordinary Python.
picker_capability_map.json, webcore_vocab.json How HA entities become webCoRE-shaped devices and attributes.
routing_table.json Which webCoRE features force the PyScript band instead of YAML.
templates/compiler/**/*.j2 The actual shape of the emitted automation / Python module.

Fixing one with an AI — the intended workflow

  1. Open Diagnostics, find the failing piston.
  2. Click Copy AI repair prompt. That copies the exact error, the piston's JSON (the compiler's input), the code the compiler produced, and the current contents of the file that governs this class of error, with instructions to reply with an edit in that file's existing shape.
  3. Paste it into any AI chat. Because the file travels with the question, the answer is a concrete edit rather than advice.
  4. Apply the edit to that file, redeploy PistonCore, and re-save the piston. Compile again — the mapping now exists.
  5. If the fix needs the compiler's Python rather than a data file, the prompt says so plainly. That one's a code change worth reporting.

This is the maintenance model on purpose: as Home Assistant grows new services and you meet more webCoRE features, the compiler is extended by adding lines to data files — not by waiting for someone to rebuild it.

Situation 4 — it compiles, runs, and does the wrong thing

A green pill means HA accepted the automation, not that the logic is right. Two things slip past every check: a template that only fails when it runs, and logic that's valid but wrong.

  • YAML pistons: Home Assistant → Settings → Automations → PistonCore: <piston name> → Traces. That shows the exact path taken: which trigger fired, what each condition evaluated to, where it stopped.
  • PyScript pistons: they don't appear in the automations list — their activity goes to HA's log, one line per wake, tagged with the piston id.
  • Compare that against View generated code. If the automation did what the code says but not what the piston means, that's a compiler bug — use Copy AI repair prompt, or report it with Copy debug info.

What each status means

StatusMeaning
deployed Compiled, written, accepted by HA's config check, and verified present after the reload. The only green.
compile error Read the message — it names the exact statement. Your piston is safe and the previously deployed version keeps running.
needs PyScript Valid piston, but it needs the PyScript integration installed in Home Assistant.
paused You paused it; its automation is disabled in HA.

Safety facts worth knowing

  • A failed compile never loses your piston and never touches the working version already running in HA.
  • If a new compile fails HA's config check, the previous good file is restored automatically before HA reloads anything.
  • Pause always works: if HA can't be reached to disable the automation, the file is removed instead. The message says which happened.
  • The compiler never writes into your piston — the JSON you author is read-only to it, always.
  • Every compile's output is kept (last ten per piston), so the evidence survives several re-saves.
