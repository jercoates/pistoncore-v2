Extending the compiler (with an AI)

PistonCore compiles your pistons using editable data files, not baked-in code. When a piston fails because the compiler doesn't know a command, a value, or a device attribute yet, you can teach it — usually by adding a few lines to a JSON file — and the fix takes effect on the next compile. No rebuild, no restart, no waiting for a new release. This page is the walkthrough; hand it to any AI along with the repair bundle and it can make the edit for you.

The fastest path

  1. Open Diagnostics and find the failing piston.
  2. Click Copy AI repair prompt. That bundles the exact error, the piston, the code the compiler produced, and the current contents of the file that governs this class of error.
  3. Paste it into any AI chat, add a link to this page, and ask it to give you the edited file.
  4. Save the AI's version over the file it names (see “Where the files live” below), then re-save the piston. It compiles again.

Where the files live

The editable copies are on PistonCore's data volume, under a customize/ folder. Diagnostics shows the exact path for your install (System → “Editable compiler files”). The layout mirrors the project:

FileWhat editing it does
templates/compiler/yaml/classic/command_maps.json Maps a webCoRE command to a Home Assistant service (turn on, set level, play track, lock, arm…). The most common fix.
templates/compiler/yaml/classic/value_maps.json Maps attribute values (motion active → on), alarm commands, thermostat/fan modes, colour names.
routing_table.json The list of features that force a piston onto the PyScript band. Remove a signature here if Home Assistant has since gained native support for it.
templates/compiler/yaml/classic/*.j2 The actual shape of the emitted automation / script YAML. Advanced — change how output looks, not just what maps to what.
templates/compiler/pyscript/2.x/expr_runtime.py.j2 The webCoRE expression functions, as Python. Add a missing function here (and to _FUNCTIONS in the code) so expressions using it compile.
picker_capability_map.json, webcore_vocab.json How Home Assistant entities become webCoRE-shaped devices and attributes. Edit if a device's attribute isn't recognised.

These are seeded from PistonCore's defaults the first time it runs and are never overwritten afterward — your edits always survive an update. Any file you delete falls back to the built-in default, so a bad edit is always one deletion away from being undone.

Instructions for the AI making the edit

If you are an AI following this page, these are the rules:

  • Keep the file valid and in its existing shape. JSON files stay valid JSON; template files stay valid Jinja2/Python. Match the surrounding style exactly — don't reformat the whole file.
  • Add, don't restructure. The goal is almost always one new entry (a command mapping, a value, a function, a removed routing signature), not a rewrite. Change the least that fixes the error.
  • command_maps.json values are either a plain "domain.service" string, or {"service": "domain.service", "data": {…}} where data values may use $1/$2 for the task's parameters, optionally with a transform ($1|hex_rgb, $1|pct_float). Look at the existing entries and copy the pattern. Use only real Home Assistant service names.
  • routing_table.json lists webCoRE-JSON signatures that force PyScript. To let something compile to a native automation instead, remove its entry — but only if Home Assistant genuinely expresses it; if unsure, leave it (PyScript is the safe fallback).
  • expr_runtime.py.j2 functions: implement def _fn_<name>(...) matching the others, and add the bare name to _FUNCTIONS in shim/compiler/expression.py so the parser accepts it.
  • Reply with the edited file (or the exact edited section) and say which file it goes in — nothing else is needed.
  • If the fix needs the compiler's Python rather than one of these data files, say so plainly. That's a code change, not a data edit, and should be reported as a bug rather than worked around here.

Checking your edit worked

  1. Save the edited file to its customize/ location.
  2. Re-save the piston (open it and save, or Enable it) — that recompiles.
  3. On Diagnostics, the piston's status should go green; View generated code shows exactly what the compiler now produces, so you can confirm the mapping did what you intended.
  4. If it's wrong, either fix the file again or delete it to fall back to the default and start over.

When Home Assistant changes under a working piston

If a piston that used to work stops — a service was renamed, a domain changed — the compile error names the Home Assistant domain and service involved. Check Home Assistant's own release notes / breaking-changes for that domain to find what changed, then update the mapping in command_maps.json the same way. PistonCore can't predict future Home Assistant changes, but it always tells you exactly where to look.
