Playing media files (Play track)

A piston's Play track action tells a speaker to play a sound file. Home Assistant plays audio files all day — chimes, sirens, alerts — so this always works; the only thing that ever needs setting up is telling HA where the file is. You do that just by how you write the URL. There is no mode to flip — each Play track URL is handled on its own, and you can mix them freely in the same piston.

Why this needs any setup at all

On Hubitat, x-file-cifs://… "just worked" because the hub itself read the file off your share and fed it to the speaker — the hub was the middleman, always on. Networked speakers (Sonos, Chromecast) never read a file share directly; they fetch a URL. Home Assistant does the exact same middleman job — it just needs to know where the file lives. That's the whole gap, and PistonCore closes it two ways.

Two ways to point at a file — pick per URL

Write the URL as…What happens
/local/Music/Siren.mp3 The file lives in Home Assistant. Drop it into HA's config/www/ folder (here, config/www/Music/) and reference it with /local/…. HA serves it. Nothing else to set up — this works on every HA install.
x-file-cifs://192.168.1.208/PiShare/Music/Siren.mp3 Old-school — the file stays on your network share. The x-file-cifs:// form tells PistonCore this is a Hubitat-style path and to make it work: it streams the file straight off the share through the PistonCore media server (set that up once, below). Your existing Hubitat pistons keep their URLs unchanged.

A full http://… URL and HA's media-source://… paths also pass straight through. Because it's decided per URL, one piston can play a doorbell chime from HA's folder and a siren from a share at the same time.

Setting up the HA folder way

  1. Put your sound files in Home Assistant's config/www/ folder. Keep them organized in subfolders if you like — a good habit is a config/www/pistoncore/ folder just for automation sounds.
  2. In Play track, reference the file as /local/… — for config/www/pistoncore/siren.mp3 that's /local/pistoncore/siren.mp3.
  3. Save the piston. That's it — HA plays it.

Home Assistant has no small file-size limit the way Hubitat did, so you can keep everything in one HA folder if you want — you no longer have to scatter large files across shares.

Setting up the old-school share way (PistonCore media server)

This lets your existing x-file-cifs:// pistons play without moving a single file. In Settings → Playing media files, turn on the media server: give it the address other devices reach PistonCore at (so speakers can fetch from it), and a login for each share it should read. PistonCore then serves those files to your speakers exactly like the Hubitat hub did.

One trade-off, stated plainly: with the media server, a share file only plays while PistonCore is running — it's the middleman. If you ever remove PistonCore, those files stop playing until you move them into Home Assistant and switch those URLs to /local/…. Files that already use /local/… never depend on PistonCore.

A note on Amazon Echo / Alexa

Echo/Alexa speakers cannot play an arbitrary sound file — Amazon blocks it, and it blocked Hubitat too. This isn't something HA or PistonCore can change. Sonos, Chromecast, VLC and most media players play files fine. If a Play track aimed at a mixed group seems to do nothing, check whether the target is an Echo.

If a Play track URL is flagged

If a piston uses an x-file-cifs:// URL and the media server isn't set up yet, the front-door status and the piston's own screen say so. Fix it by either turning on the media server, or moving the file into HA and using a /local/… URL. Everything else about the piston still deploys.
