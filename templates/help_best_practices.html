Getting started with PistonCore

PistonCore gives you the real webCoRE editor on top of Home Assistant. You write pistons exactly the way you always have; PistonCore turns each saved piston into native Home Assistant automations. This page covers the habits that make that go smoothly.

Where things live

The front door (PistonCore's own list) is home base: every piston, its compile status, whether it's running. Clicking a piston opens the webCoRE editor — the same editor, unchanged — and leaving it brings you back. Import/Export and Settings are PistonCore pages; everything piston-shaped is webCoRE's.

Bringing pistons in

  1. Set up your global variables first. A piston that uses @Speakers or similar refers to it by name only — the devices live in your instance, not in the piston. Create the globals with the same names (they're case-sensitive) and assign devices before compiling, or the piston will report the missing global.
  2. Import through the Import page (paste JSON) or the + New Piston dialog (restore code, backup file, duplicate). Either way webCoRE's own device-matching step runs, so imported pistons bind to your real devices — usually automatically, by name.
  3. Save once, then check the front door. Compiling happens on save; the status pill tells you what happened.

What the status pills mean

PillMeaning
deployed Live in Home Assistant and verified present after the reload.
compile error Something in the piston can't be built yet, or a device/global is missing. The message names the exact statement — and your piston is safe; the previously working version keeps running.
paused You paused it; its automation is disabled in HA.

Full detail, and what to collect when something goes wrong: Debugging a bad compile.

Install PyScript — recommended

Simple pistons become ordinary HA automations. Anything richer — formulas, switch, loops, variables, event blocks, computed notification or speech text — compiles to PyScript instead. Without the PyScript integration those pistons will report that they need it. Install it once from HACS (search “pyscript”) and every kind of piston can run. PistonCore tells you which band each piston used.

Habits that keep things predictable

  • Pause instead of deleting while experimenting — pausing disables the automation in HA and keeps its history; enabling recompiles it fresh.
  • Let PistonCore own its files. Everything under pistoncore/automations/ and pyscript/scripts/pistoncore/ is regenerated on every save — edit the piston, never the generated file.
  • Rename freely. Renaming a piston moves its compiled file cleanly; no orphans left behind.
  • Use Home Assistant's own traces when a piston runs but does the wrong thing: Settings → Automations → your piston → Traces shows the exact path it took.
  • Keep credentials current. If you regenerate your HA token, update it in Settings — the front door's health badge is the fastest way to spot a broken connection.

If you get stuck

Every compile keeps its output. The three things that make any problem diagnosable — the piston's exported JSON, the newest file in its compile_debug folder, and the banner message — are described in Debugging a bad compile. You can re-run setup any time from Settings.
